# US-10: Frontend unit tests for `TimingWaterfall`

**Feature:** 0019 - Detailed Request Timing Breakdown
**Status:** [ ] Pending
**Dependencies:** US-6

---

## Description

As the frontend test suite, I want Vitest unit tests for the `buildPhaseRows` pure function and the rendered `TimingWaterfall` component so that correctness of offset computation, conditional TLS row, and connection-reuse display are verified automatically.

---

## Acceptance Criteria

**`buildPhaseRows` unit tests:**
- [ ] TLS row is omitted when `tlsHandshake === 0`.
- [ ] TLS row is present when `tlsHandshake > 0` (HTTPS scenario).
- [ ] `offsetMs` for each row equals the cumulative sum of all preceding rows' `durationMs`.
- [ ] All five rows are returned for a full HTTPS timing object with no zero values.
- [ ] Returned array length is 4 (no TLS) or 5 (with TLS).

**`TimingWaterfall` render tests:**
- [ ] "Connection reused" note is rendered when `dnsLookup === 0 && tcpHandshake === 0` and total > 0.
- [ ] "Connection reused" note is NOT rendered when `dnsLookup > 0`.
- [ ] Each bar element has an `aria-label` containing the phase name and duration.
- [ ] Tooltip content (via `title` attribute or data attribute) includes start offset and duration for each phase.
- [ ] Total row displays the correct sum of all phases.
- [ ] "No timing data available" message is rendered when all phase values are `0`.
- [ ] "Send a request to see timing data" placeholder is rendered by `ResponseViewer` when `timing` is `undefined` (integration boundary test — may be placed in `ResponseViewer.test.tsx` instead).

**General:**
- [ ] All tests pass with `npm run test`.
- [ ] No snapshot tests — assert on specific DOM content and attributes.

## Notes

- Use `@testing-library/react` (`render`, `screen`, `getByRole`, `queryByText`) consistent with existing frontend test patterns.
- Mock `TimingPhases` fixtures should cover three scenarios: full HTTPS, HTTP (no TLS), and connection-reuse.
- Do not assert on exact CSS class names or pixel widths — assert on semantic content (`aria-label`, text content, presence/absence of elements).
- Keep tests independent: each `it()` block renders fresh with its own fixture — no shared mutable state.
